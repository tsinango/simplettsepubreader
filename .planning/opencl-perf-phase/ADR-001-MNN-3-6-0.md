# ADR-001: 将 MNN 子模块升级至 3.6.0

**状态**: 已通过 | **日期**: 2026-06-23 | **决策者**: tsinango

## 背景

项目 pinned 于 MNN v3.5.0（提交 `8d9f613`，"Support low-bit ARM quant kernels"）。
BV2 OpenCL 推理速度较慢，且所有测试模式（OPENCL_ALL/OPENCL_DECODER/OPENCL_FLOW）
使用 BUFFER+IMAGE、FAST/WIDE 调优均未击败 CPU 基线。

用户提到 MNN 3.6.0 新增了 OpenCL GEMM/GEMV 启发式分布，
可能直接影响 flow transformer 中的卷积 kernel。

## 决定

将 `third_party/MNN` 从 `8d9f613`（v3.5.0）升级至最新可用的 v3.6.x 标签。

## 理由

- 3.6.0 因对 OpenCL 后端中 GEMM/GEMV kernel 的改进而被提及
- 在 3.5.0 上继续微调调优参数（FAST→WIDE→HEAVY）对 kernel launch 开销无影响
- 升级风险低：MNN API 保持稳定的 Express+Module 接口；若失败可回滚至原提交

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| API 漂移导致 `bert_vits2_v23_loader.cpp` 编译失败 | 保留 `8d9f613` 回退指针在 `.gitmodules` 中；若失败则还原 |
| MNN_GPU_TIME_PROFILE 路径修改 | 构建后验证日志输出；若 deviated 则调整 patch |
| 构建系统更改（CMake flag deprecation） | READ BUILD-REPRODUCIBLE.md 并适配 |

## 验收标准

1. `build_mnn.sh opencl` 无误完成
2. 生成的 `libMNN.so` 与已存在的 `bertvits2_jni.cpp` 无符号错误链接
3. 设备端 `logcat MNNJNI:I` 仍可见 `kernel time =` 行
4. `bv2RunBenchmark` 无崩溃运行至少 10 次推理
